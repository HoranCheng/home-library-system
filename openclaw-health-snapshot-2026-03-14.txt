  不是当前 workspace 里单独注册成独立 agent 的常驻条目；更多是 ACP 允许的外部 coding agent / provider 路由目标，用于 Claude Code / Anthropic 模型链路。
- chatgpt:
  也不是当前 workspace 里单独注册成独立 agent 的常驻条目；更准确说是 OpenAI Codex / GPT-5.4 这条模型链路，被用作 fallback，或者某些频道/session 直接运行在 gpt-5.4 上。
- 其他:

  但 agents_list 对当前 requester 真正暴露的本地 agent 只有 main。
  另外运行中还能看到 subagent / cron / heartbeat 这类 session 形态，不等于独立人格 agent。

2. 当前模型路由 / fallback
- 默认模型:
  anthropic/claude-sonnet-4-6
- Claude 什么时候用:
  默认就是先走 Claude Sonnet 4.6；当前频道 session_status 也显示主链路是这个。
- ChatGPT 什么时候用:
  当主模型不可用、限流、认证失败，或者某些 session 被直接切到 gpt-5.4 时使用。
  现在已有多个 Discord 会话直接在 gpt-5.4 上跑。
- retry 逻辑:
  openclaw.json 里没有显式配置 retry 次数。
  但从这次真实 transcript 观察到的行为是：Claude 先多次重试 429，之后出现 401，再切到 gpt-5.4 fallback。
  所以可以理解为: 先在 primary 上做内部重试，再落到 fallback；具体重试次数看实现，不是你当前 config 里单独写死的字段。

3. system / handover 大概怎么写
- 主 prompt 长度大概:
  偏长，不是轻量 prompt。
  由 system + developer + skills 规则 + workspace context (AGENTS/SOUL/USER/HEARTBEAT 等) + inbound metadata 组成。
  从这次进入 gpt-5.4 的首轮输入看，单轮输入量大约已经在 1.3 万 token 级别，还没算后续工具结果继续堆进去后的增长。
- handover 是否带全文历史:
  正常 session 会带该 session 的历史，不是“每次只给最后一句”。
  但不是无限原样全文保留：上下文接近上限时会 compaction / pruning。
  跨 agent / 跨 session 不会天然共享全部全文，除非通过 thread、transcript、attachments、memory、明确 handoff 去传。
- 是否带工具结果:
  会带，但会被裁剪。
  尤其长 tool result 会进入 context pruning 规则：soft trim 或 hard clear，而不是永远完整塞回。

4. summary / 压缩机制
- 什么时候触发:
  你这里启用了 compaction.mode = safeguard。
  配置里 reserveTokensFloor = 76000，说明接近上下文压力时会触发压缩保护。
  另外对工具结果还有独立 contextPruning 机制，不用等到真正 compaction 才会裁剪。
- 压缩后原文是否还保留:
  在 transcript / session 文件层面通常还在；但喂给模型的活跃上下文不一定全保留。
  对超长工具结果，hardClear 之后在上下文里会被替换成占位符: [Old tool result content cleared]
- 谁来做 summary:
  不是你手工写。
  是 OpenClaw/runtime 触发的模型侧 compaction / pruning 机制来做。
  另外你还开了 memoryFlush：临近 compaction 时会提示把 durable notes 写进 memory/YYYY-MM-DD.md。

5. 工具结果处理
- exec 输出会不会整段塞回:
  小输出通常会跟进会话；大输出不保证完整保留。
  实际上它会先进入 transcript，但在继续对话时可能被 prune / trim。
- 长日志会不会截断:
  会。
  明面上至少有三层截断/压缩：
  1) 工具本身的返回限制，例如 read 工具上限 2000 行或 50KB；
  2) transcript / UI 展示也会局部截断超长内容；
  3) contextPruning 对长 tool result 做 softTrim / hardClear。
  你当前 softTrim 配置是 maxChars 3000，head 1200 + tail 1200；minPrunableToolChars = 30000。
- 错误日志如何处理:
  会进入 transcript，能被后续分析到；必要时也会被当作 toolResult / assistant error message 留档。
  但同样可能因上下文修剪被缩短，不等于永远原样全量保留在活跃上下文。

7. errors 大概构成
注意: 下面不是严格监控报表，是我基于 session transcript + memory + 本次现场观察给的实战近似。
- rate limit:
  很显著，尤其 Anthropic 这条链路。你这次请求本身就现场复现了连续 429。
- timeout:
  也很多。日志粗扫里 timeout 相关痕迹很多，常见于 browser tool、外部工具、长任务、网页自动化。
- model unavailable:
  有，但通常少于 rate limit / timeout。包括认证异常、provider 不可用、invalid key、模型链路切换。
- tool fail:
  有一批。典型是 browser tool timed out、unsupported_country、ACP auth 问题、无 accessibility 权限、子代理 accepted 但没产出。
- 其他:
  还有“任务 accepted 但无输出”“配置/权限边界不一致”“多会话把 Claude 配额吃光”这类不是单纯 API error code 的系统性问题。
