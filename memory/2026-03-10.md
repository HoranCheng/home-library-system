## Library System — Cover Uniformity, Zoom Lock, Versioning (evening session)

### Cover Image Fixes (commits 82d4b2e → 32d5b00)

**Problem**: Book covers had wildly inconsistent sizes; "image not available" from Google Books treated as real covers.

**Fix 1 — Uniform sizing (bf0c318)**:
- `aspect-ratio:2/3` on `<img>` doesn't reliably work on iOS Safari
- Replaced with classic **padding-top:150% container** trick (`.cover-wrap`)
- `<img>` and `.cover-placeholder` are `position:absolute` inside, filling the container
- Works on all browsers

**Fix 2 — Google Books placeholder detection (32d5b00)**:
- Google Books "image not available" placeholder: exactly **128×196 px**, ~6-11KB
- Old detection threshold (<=80×120) missed it completely
- New: any `books.google.com` image <=200px wide → treated as placeholder
- Volume IDs ending with `ACAAJ` → flagged as no-real-cover at source level
- `isGBPlaceholderUrl()` filters in both `parseGoogleBooks()` and `extractAnyCover()`

**Fix 3 — Bookcover API integration (df47727)**:
- **https://bookcover.longitood.com/bookcover/{ISBN}** — free, no API key, returns Goodreads/Amazon cover URLs
- Cover source priority during ISBN lookup:
  1. Google Books thumbnails (filtered for placeholders)
  2. Bookcover API (best coverage)
  3. OpenLibrary ISBN cover (last resort)
- Tested 7 ISBNs that had no covers: **7/7 found by Bookcover API**

  - 9781761065149, 9781760556457, 9780349432120, 9780500653623
- One-time migration (`lib:cover-migration:v2`): fetches covers for books with missing or GB-placeholder URLs
- `coverUrl()` function simplified: only returns stored `coverUrl`, no more OL 1x1 GIF fallback

### ISBN Lookup Speed Optimization (commit 7b33892)

**Problem**: 8 API requests fired in parallel, waited for ALL to complete → 5-7 second perceived wait

**Solution: 2-phase lookup**

Phase 1 (fast, ~300-800ms):
- Google Books (via Worker cache) + Bookcover API fire in parallel
- If title + author found → form fills IMMEDIATELY

Phase 2 (background, non-blocking):
- Open Library runs silently to fill empty fields (category, year, cover)
- User never waits for this

Slow fallback (only when Google Books has no data):
- All 6 sources queried in parallel
- Shows "正在查询更多数据源…"

**Result**: Most books go from 5-7s → under 1s

## Library System — Security + Product Audit Report (late night)

### Full audit completed and documented
- Reviewed `library-system/preview/index.html`, `worker/books-proxy.js`, deployment artifacts, third-party dependencies, and product positioning
- Wrote full report: `library-system/docs/SECURITY_PRODUCT_AUDIT_2026-03-10.md`
- Committed and pushed report: `37ad22a` (`docs: add security and product audit report`)

### Most important findings
- **P0:** Cloudflare Worker CORS origin check is vulnerable because it uses `startsWith()`; confirmed malicious origin like `https://horancheng.github.io.evil.com` is accepted in `Access-Control-Allow-Origin`
- **P0:** Worker is still a publicly callable proxy even if CORS is fixed; needs rate limiting / parameter allowlist / maxResults clamp / caching to prevent Google Books quota abuse
- **P0:** GitHub Pages deployment is incomplete: verified live 404s for `/manifest.json`, `/sw.js`, `/icon-192.svg`, `/icon-512.svg`
- **P0:** `makeNotice()` builds HTML directly and some call sites pass unescaped `title` / `e.message`, leaving an XSS path via imported or malformed data
- **P1:** Service worker caches all GET requests too broadly
- **P1:** No CSP
- **P1:** Import validation is shallow; needs schema/length/url checks
- **P1:** Source/deploy structure is getting messy (`preview`, `library-system/docs`, root `docs`)

### Product/market conclusion
- Overall direction is strong: Chinese home-library management, mobile/PWA-first, good scan UX, and lightweight/no-login setup are real strengths
- Main near-term risk is not account takeover but API quota abuse, broken deployment polish, and frontend injection surfaces
- Best strategic positioning: **be the best Chinese household physical-book catalog app**, not Goodreads/social reading
