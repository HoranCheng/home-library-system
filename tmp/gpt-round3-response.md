## 一、逐条评估 Claude 的新发现

## S1：CSP 允许 `unsafe-inline`，XSS 防护有限（Claude 标 P1）
**我的结论：认同事实，不认同优先级。**

### 我为什么这么看
当前 `preview/index.html:11` 确实是：
- `script-src 'self' 'unsafe-inline' https://unpkg.com`

这意味着 CSP 不是“强防线”，只能算一个很弱的兜底。
但这不是一个新的可直接利用漏洞，它更像是：
- 因为项目是单文件 app，架构上离不开 inline script
- 所以 CSP 现在天然打折

### 优先级判断
- **不是 P1**，更像 **P2 / 技术债**
- 前提是：当前已知注入点先收干净
- 如果当前还有真实 XSS 面，那它才会被放大

### 建议
- 近期不用为了这个大拆文件
- 但长期如果继续做账号同步和云功能，还是应该把 JS 外提，逐步摆脱 `unsafe-inline`

---

## S2：`coverUrl` 多处直接进入 `img src`，缺协议校验（Claude 标 P1）
**我的结论：问题存在，但优先级偏高了。**

### 代码情况
`coverHtml()`、`showSuccessModal()` 等地方确实把 `coverUrl` 直接塞进 `<img src>`。
虽然有 `escapeHtml()`，但那不是协议白名单。

### 风险判断
这个点我认为是：
- **防御性修复项**，不是当前主要安全洞
- 现代浏览器里 `img src="javascript:..."` 通常不会执行
- 真正风险更多在于脏数据、异常 URL、奇怪 data URI 之类的边界情况

### 优先级判断
- **P2 更合适**，不是 P1

### 建议
- 在统一的 `coverUrl()` / `isSafeUrl()` 逻辑里做协议收口
- 允许 `https:`，必要时允许 `http:`
- 不建议继续放开任意 `data:`

---

## S3：CSP 的 `connect-src` 不完整，会拦住 Bookcover / OpenLibrary / Crossref（Claude 标 P2）
**我的结论：认同，而且优先级应该比 Claude 标得更高。**

### 这是个真 bug
当前 CSP `connect-src` 只有：
- `'self'`
- Worker 域名
- `https://www.googleapis.com`

但代码里实际还有直接请求：
- `https://bookcover.longitood.com/...`
- `https://openlibrary.org/...`
- `https://api.crossref.org/...`

这不是理论问题，是**配置和运行时行为不一致**。
如果 CSP 生效，这些请求就是会被拦。

### 优先级判断
- 我认为这不是 P2，应该是 **P1 功能 bug**
- 因为它会直接影响查书、补元数据、封面获取

### 建议
- 立即把这几个域补进 `connect-src`
- 顺手检查 `img-src` 是否覆盖了 OpenLibrary 封面域（目前 `img-src https:` 倒是够）

## S4：前端删除书籍只做本地移除，不发 tombstone（Claude 标 P1）
**我的结论：完全认同，而且这是这轮最重要的新问题。**

### 这条我确认了，Claude 说得对
当前删除逻辑还是：
```js
const books = loadBooks().filter(b => b.id !== id);
saveBooks(books);
```
也就是：
- 本地直接删掉对象
- `saveBooks()` 触发 push
- push 发送的是“当前 books 全量”
- 被删掉那本已经不在 payload 里了

而服务端 `sync.js` 已经支持 `_deleted` tombstone。
问题出在前端根本没把删除当成同步事件发出去。

### 后果
Claude 这里判断是对的：
1. 其他设备不知道这本书被删了
2. 云端还保留旧记录
3. 后续 pull 时，本地删除可能被“复活”

### 优先级判断
- **P1 是合理的**
- 如果把云同步当正式功能看，甚至可以说接近 **P0.5**

### 修复要求
这里不能只做一个小补丁，必须一起改：
1. 删除时改成：`_deleted: true, updatedAt: now`
2. 前端渲染时统一过滤 `_deleted`
3. pull merge 继续保留 tombstone 语义
4. 初始同步、冲突处理都要按“删除也是变更”来算

这条是我认为 Claude 最值得立即推动修的点。

## B4：`metadataSources` 类型不一致（前端字符串，导入校验/服务端按数组思路）
**我的结论：认同，而且这是 Claude 这轮抓得比较准的一个数据一致性问题。**

### 我确认了
当前前端保存时还是：
```js
let metadataSources = String(els.manualCard.dataset.bookSources || '').trim();
```
也就是字符串。

但：
- 导入校验里对 `metadataSources` 非数组会清成 `[]`
- 服务端 `sync.js` 是 `JSON.stringify(book.metadataSources || [])`
- 服务端取回又 `tryParseJson(..., [])`

### 这会导致什么
- 同步前是字符串
- 服务端落库时会变成 JSON 字符串
- 再拉回来时可能是字符串而不是预期数组结构
- 数据语义混乱

### 优先级判断
- **P1.5 / P2 之间**
- 不像删除 tombstone 那么急，但确实应该尽快统一

### 建议
- 统一成数组，前端展示时 `join(' / ')`
- 别再把它当自由字符串用

## 二、Claude 可能遗漏的问题

我认为 Claude 这轮有两个值得补充的点：

## 1）删除问题不只是“前端没发 tombstone”，还包括“前端本地模型里没有保留已删记录”
Claude 已经抓到核心 bug，但我补一句：

如果前端继续采用“本地直接 filter 掉”的模型，那么：
- 即使服务端支持 tombstone
- 前端也没有地方保存本地删除事件
- 这会让删除传播、离线删除、冲突比较都不完整

所以真正该修的是：
- **把删除从“数组移除”改成“状态变更”**
- 也就是本地数据模型正式支持 `_deleted`

这点比 Claude 原文更关键。

---

## 2）`metadataSources` 不一致不仅是展示问题，也会污染同步语义
Claude 提到了类型不一致，但没把影响说透。

这不只是“字段格式不优雅”，而是会造成：
- 前端、导入逻辑、服务端存储对同一字段的理解不同
- 之后任何 merge / 导出 / UI 展示都可能继续分叉

这条应该被看作**数据模型一致性问题**，不是普通 UI 小问题。

## 四、我认为 Claude 应该立即修的清单

下面这份是我认可的“这轮立刻修”清单，按优先级排序：

## 第一优先级（必须立刻修）
1. **删除不发 tombstone**
   - 删除改成 `_deleted: true`
   - 渲染过滤 `_deleted`

   - 不要只 `auth.clear(); render();`

6. **批量操作增加确认框**

## 五、最终结论

这轮 Claude 的新扫描里，**最有价值的新增发现其实就三条**：

1. **删除不发 tombstone** —— 真同步 bug，而且现在还没修
2. **CSP `connect-src` 不完整** —— 真功能 bug，会影响封面/元数据请求
3. **401 过期提示缺失** —— 真体验断裂，会让同步看起来像莫名失效

另外两条值得尽快顺手修：
- `metadataSources` 类型统一
- 未登录时“毛毛的书房”改成“我的书房”

至于：
- `unsafe-inline`
- `coverUrl` 协议校验
- 大量 `innerHTML`

这些我认同是长期风险/技术债，但**不该排到这轮最前面**。先把“会出错、会混乱、会影响同步正确性”的东西收掉，更值。
