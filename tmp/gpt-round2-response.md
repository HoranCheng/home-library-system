## P0-2：XSS + JWT localStorage
**结论：部分同意，当前方案不完整。**

Claude 说：
1. `makeNotice()` 默认 escape
2. 全文搜索 `innerHTML`
3. JWT 缩短 TTL 到 4 小时 + refresh token
4. 长期加 CSP

其中 **1 和 2 是必须立刻做的**，但 **3 和 4 的表述还不够严谨**。

### 我同意的部分
- `makeNotice()` 默认 escape：对
- 全面排查 `innerHTML`：对
- CSP 要加：对

### 我不同意/要补强的部分
1. **不能把问题表述成“修 makeNotice 就够了”**
   - 原问题不是只有 `makeNotice`
   - 是“动态内容进 innerHTML 的体系性风险”
   - 所以修复标准应该是：
     - 能用 `textContent` 的地方全部改 `textContent`
     - 必须 `innerHTML` 的渲染点，明确只接受模板+escape 后变量

2. **CSP 不能只说“长期加”，至少这轮要加基础版**
   - 因为这是很低成本、收益很高的防线
   - 哪怕先上一个保守版，也比没有强
   - 如果页面强依赖内联脚本，说明架构上本来就有债，那就把 CSP 调整成当前可落地的最低可用版本，而不是继续空着

3. **“PWA 没有 httpOnly cookie 选项”这个说法不严谨**
   - 不是 PWA 不能用 cookie，而是你当前部署形态和接口形态没有做成那套
   - 这点别说错，不然容易把技术限制归因错了

4. **refresh token 不是一句话就算方案**
   - 如果这轮不做 refresh，就明确写“本轮不做 refresh，只缩 TTL”
   - 如果这轮做 refresh，就必须包含：
     - refresh token 存储位置
     - 轮换策略
     - 服务端吊销或最少版本控制
   - 否则只是把复杂度写进计划，没有真正形成修复方案

### 我的建议
- 本轮最低要求：
  1. `makeNotice` 默认 escape
  2. 扫清所有危险 `innerHTML`
  3. 所有错误消息展示统一走安全文本渲染
  4. 加基础 CSP
  5. access token TTL 缩短
- refresh/revoke 可以后补，但不能拿它替代这轮该做的 XSS 收口

### 结论
- **同意立即修，但要求把“全面消灭危险 innerHTML + 基础 CSP”纳入本轮**

## P0-4：Books 代理无防护
**结论：基本同意，但还少两刀。**

Claude 给的方案：
- 参数白名单
- `maxResults` clamp
- IP 限流
- 常见 ISBN 查询加 Cache-Control

这方向是对的，但还不够完整。

### 要补充的点
1. **不只是 `maxResults`，还要限制 `q` 长度和 `startIndex`**
   - 否则还是能被人拿来跑重查询或深翻页
   - 建议：
     - `q` 长度上限
     - `startIndex` 最大值限制，或者直接不开放

   - 要确认这个限流是否适合匿名公共代理
   - 登录接口限流和 books 代理限流不一定应该共用同一配额
   - 最少也要分开 key namespace

3. **最好只允许你实际前端会用到的查询模式**
   - 如果前端主要是 ISBN 查询，就别把整个 Google Books 搜索能力开放成通用代理
   - 范围越窄越安全

4. **缓存策略最好明确到 ISBN 场景**
   - “常见 ISBN 查询加 Cache-Control”太泛
   - 更实用的是：对 ISBN 精确查询做缓存，对自由文本搜索保守处理

### 结论
- **同意立即修**
- **补充要求：限制 `q` / `startIndex`，并尽量把代理收窄成前端真实需要的最小能力**

## P1 项逐条意见

## P1-1：删除复活 Bug
**结论：认同问题，Claude 的修法方向对，但还不完整。**

他说：
- full pull 也返回 tombstone
- 前端 merge 处理删除
- push 时不再推送云端已删书

这只是修了一半。

### 缺的关键点
1. **tombstone 不能只在 full pull 返回，还要有稳定的时间戳语义**
   - 删除本质上也是一次变更
   - tombstone 必须带 `updated_at` / `deleted_at`
   - 否则合并时无法判断新旧

2. **前端不能“直接从本地删除”这么粗暴**
   - 如果本地存在比远端 tombstone 更新的修改，应该进入冲突判断
   - 否则会出现“本地刚改，远端旧删除覆盖本地新改”

3. **push 端必须显式传删除状态，而不是简单忽略已删书**
   - 如果本地也允许删除，那 push 时必须能把 tombstone 往云端发
   - 否则删除传播仍然是不完整的

4. **initial sync 逻辑要重写，不是小补丁**
   - 原问题核心是“首次 pull 后再全量 push”
   - 所以要检查 initial sync 是否还在做危险的全量回推
   - 如果还在，这个 bug 迟早以别的形式回来

### 我的判断
Claude 已经看到了问题，但他写的方案还是偏“把 full pull 补点 tombstone 就结束”。这不够。

### 结论
- **同意修，但要求：把删除视为正式同步事件，补齐 tombstone 时间戳、merge 规则、push 传播、initial sync 重构**

## P1-2：冲突未处理
**结论：部分同意，当前方案太简化。**

Claude 方案：
- push 返回 conflicts 后自动 pull
- 提示“X 本书云端有更新版本，已同步最新”

这个做法比现在强，但有一个问题：

### 我不同意的点
**“已同步最新”这个文案不准确，甚至误导。**
- 真相不是“同步成功”
- 真相是“发生冲突，本地版本未被接受，已回拉云端版本”
- 这两者不是一回事

### 需要补充的点
1. **提示必须明确是冲突，不是成功**
   建议类似：
   - “3 本书与云端版本冲突，已保留云端较新版本”
   - 或 “检测到 3 条冲突，本地修改未上传，请检查”

2. **最好记录冲突书 ID 或列表**
   - 不一定马上做 UI，但至少日志/状态里要留痕
   - 否则用户根本不知道是哪几本

3. **自动 pull 可以做，但别静默吞掉本地修改**
   - 如果策略就是 newer-wins，那要明确告知
   - 不能假装什么都没发生

### 结论
- **同意立即修**
- **但提示文案和状态语义必须改，不能伪装成普通同步成功**
