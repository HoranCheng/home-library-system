# Receipt Renamer 审计摘要（老板版）

## 一句话结论
这是一个方向很对、体验不错的产品雏形，但当前还不适合公开放量。最大风险不在识别效果，而在 **Worker 防滥用不足**，可能导致 API 成本损失和服务不稳定。

## 最重要的 5 件事
1. Worker 现在信任前端传来的 uid，**可被伪造绕过配额**
2. Worker 允许任意站点跨域调用，**相当于公开 AI 接口**
3. Worker 没有限制请求大小 / MIME，**可被大文件或脚本滥用**
4. 前端本地保存 token 和图片缓存，**共享设备 / XSS 时风险放大**
5. Drive 成功但 Sheets 失败时，**可能静默漏账**

## 当前状态判断
- 适合：自己用、小范围内测、继续快速迭代
- 不适合：公开放量、被大量陌生用户访问

## 你这个产品好的地方
- 拍照 → AI → 改名 → 归档 的主链路非常对
- 数据存在用户自己的 Google Drive，这点很强
- 只把低置信度异常交给用户审核，这个设计很成熟
- 移动端异步体验正确

## 你现在最该修的顺序
### 第一优先级（安全保命）
- 删除 debug 接口
- Worker 改成验证真实用户身份，不再信任裸 uid
- 限制 body size / MIME
- 收紧 CORS
- 增加真正的 rate limit / quota 防滥用
- 修自动命名逻辑，统一 merchant 名称

### 第二优先级（可靠性）
- 退出登录 / reset 时清理本地敏感缓存
- 重写 service worker 缓存策略
- 给 Sheets 增加补偿重试机制
- 修测试

## 总评价
> 这是个值得继续做的产品，但目前更像“很好的 alpha”，不是“已经安全可放量的公开产品”。
