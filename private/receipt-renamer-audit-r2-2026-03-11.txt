A3. 文件命名 — 已统一为 merchant
----------------------------------------------------------------------
原问题：processor.js（自动处理）使用 category 命名，
        ReviewView.jsx（手动审核）使用 merchant 命名，两条路径不一致。
        这直接影响文件检索价值和"识别后变成可管理资产"的主链路。

处理：
  - processor.js 命名逻辑改为：
    `${safeDate} ${safeMerchant} ${safeAmount}.${ext}`
    例：2026.03.10 Woolworths 45.50.jpg

  - safeMerchant 使用 toTitleCase()（同 ReviewView.jsx 逻辑）
    例：WOOLWORTHS METRO → Woolworths Metro

  - ReviewView.jsx 原本就是 merchant 命名（格式一致）

  修改路径：
    src/services/processor.js 第 150-152 行

  影响范围：
    - 新上传的收据会立即使用新命名
    - 已存在的文件不受影响（不追溯重命名）
    - Sheets 记录中的 merchant 字段本来就是 AI 识别的商家名，不受影响
    - 审核通过后的命名和自动处理的命名完全一致

验证：构建通过，两条命名路径现在使用相同的格式：
      `日期 商家 金额.扩展名`

----------------------------------------------------------------------
A4. 输入限制 — 全链路统一
----------------------------------------------------------------------
原问题：前端限制 20MB，AI 层限制 10MB base64（≈7.5MB 文件），
        策略不一致导致前端放行的文件到 AI 层被拒绝。

处理后的统一规则：

  阶段          限制              拦截点
  ─────────────────────────────────────────────────────
  文件选择      10MB / 文件       ScanView.jsx（alert 提示）
  压缩后        通常 <1MB         compressImage()（仅开启时）
  base64 编码   14MB base64       ai.js（兜底，10MB 文件≈13.3MB base64）
  AI 代理接收   Worker 端限制     Worker 端（需另行配置）

  MIME 白名单（全链路一致）：
  - image/jpeg ✓
  - image/png  ✓
  - application/pdf ✓

  已移除（未经全链路验证）：
  - image/webp — Canvas toBlob 输出为 JPEG，WebP 输入未验证 AI 识别效果
  - image/heic — iOS HEIC 需要额外解码库，Canvas 不直接支持

  用户体验：
  - 超过 10MB 的文件在选择时就被拦截，提示开启压缩
  - MIME 不匹配在选择时拦截，提示支持的格式
  - 不会出现"前端放行了但后端拒绝"的情况

验证：ScanView.jsx 和 ai.js 的限制值一致。构建通过。

A5. 本地敏感数据 — 全量用户隔离
----------------------------------------------------------------------
原问题：上一版只隔离了 localStorage，IndexedDB、Sheets outbox、
        SW 内存态 token 均未隔离。

处理后的完整隔离清单：

  存储位置              隔离方式                    状态
  ─────────────────────────────────────────────────────────
  localStorage          key 加 ::userId 后缀        ✅ 已完成
    rr-config           ✅
    rr-receipts         ✅
    rr-proc-progress    ✅
    rr-non-receipt-alerts ✅
    rr-sheets-outbox    ✅ （本轮新增）

  IndexedDB
    rr-image-cache      DB 名加 ::userId 后缀       ✅ 本轮新增
    rr-pending-uploads  DB 名加 ::userId 后缀       ✅ 本轮新增
    rr-sw-queue         SW 内部使用，已有 uid 字段    ⚠️ 见下方说明

  Service Worker
    _accessToken 变量   登出时通过 postMessage 清除  ✅ 本轮新增

  rr-sw-queue 说明：
  SW 的 IndexedDB 运行在 SW scope（与主线程隔离），且每个 task 已携带 uid。
  但 DB 名本身未加 userId 前缀——因为 SW 在主线程设置 rr-current-user 之前
  就可能启动。如果需要彻底隔离，需要在 SW 内部也维护用户身份状态。
  当前风险：低。SW queue 中的 task 只包含 Drive file ID 和 proxy URL，
  不包含文件内容或认证凭证。且 task 完成后自动删除。

  登出时清理流程：
  1. signOut() — 清除 Google token client 状态
  2. clearSWToken() — postMessage 通知 SW 清除 _accessToken
  3. 不删除用户数据（按 scope 隔离）
  4. 新用户登录 → setCurrentUser(newSub) → 自动切到新 scope

  工厂重置清理流程：
  1. clearAllData() — 删除所有 rr-* localStorage 条目
  2. indexedDB.deleteDatabase() — 删除三个 IndexedDB
  3. setConfig(DEFAULT_CONFIG) — 重置 React state

验证：逐一检查每个存储位置的读写路径，确认均使用 scoped key/DB name。

----------------------------------------------------------------------
A6. Sheets 写入重试 + outbox
----------------------------------------------------------------------
原问题：Sheets appendToSheet 失败时只打 console.warn，数据静默丢失。

处理：
  1. 重试：3 次指数退避（1.5s → 3s）
  2. 3 次全失败 → 写入 rr-sheets-outbox（用户隔离的 localStorage）
  3. outbox 记录格式：{ fileId, row, sheetId, failedAt, retries }
  4. retrySheetOutbox() 函数可在 app 恢复时调用
  5. 超过 10 次总重试后放弃（避免无限积累）
  6. receiptRecord.sheetSyncFailed = true 标记（UI 可显示同步状态）

验证：构建通过。outbox 读写路径使用 _getOutboxKey() 获取用户隔离的 key。

----------------------------------------------------------------------
A7. Remember Me — 移除自建 checkbox
----------------------------------------------------------------------
原问题：会议要求优先依赖 Google 原生登录保持，而不是自建"记住我"。
        上一版保留了 checkbox 只改了文案，没有真正对齐会议要求。

处理：
  1. 彻底删除 ConfigView.jsx 中的 rememberMe checkbox
  2. 所有 requestAccessToken 调用改为 persistent: false
  3. Token 只存 sessionStorage（关闭标签页即过期）
  4. 跨会话登录恢复完全依赖 Google session cookie：
     - 用户打开 app → tryRestoreSession() 尝试 sessionStorage
     - 失败 → requestAccessToken({ prompt: '' }) 利用 Google cookie 静默刷新
     - Google cookie 也失效 → 弹出 Google 登录界面

  已知限制（坦白说明）：
  - Google session cookie 有效期由 Google 控制（通常 2 周左右）
  - 用户清除浏览器数据后需要重新登录
  - iOS Safari 对第三方 cookie 限制严格，可能导致跨会话恢复失败率较高
  - 要彻底解决需迁移到 authorization code flow + 后端 token 管理
    （成本高，当前阶段不做）

  与会议要求的对齐度：
  - ✅ 已移除自建 checkbox
  - ✅ 已切换到 Google 原生 session 机制
  - ⚠️ 在 implicit flow 框架下，这是 Google 原生 remember-me 的最佳实现
  - ⚠️ 完美方案（authorization code flow）需后端支持，不在本轮范围

验证：构建通过。grep 确认无 rememberMe 状态变量。

B1. 图片压缩功能
----------------------------------------------------------------------
功能：设置页新增"上传前压缩"开关。
实现：Canvas 本地压缩，1280px 宽，JPEG 82% 质量。
效果：约 1-5MB 照片 → 200-500KB，小票文字仍清晰可辨。

     系统会自动转换为 JPEG（大多数情况）
  3. 压缩后图片为 JPEG 格式，即使输入是 PNG
     影响：PNG 格式的收据较少见，且 JPEG 压缩后仍可识别

----------------------------------------------------------------------
C1. Worker 用户身份验证 [P0]
----------------------------------------------------------------------
现状：前端通过请求体传 uid 字段，Worker 直接信任。
风险：任何人可以伪造 uid，冒充其他用户消耗配额。

关于方案选择——ID token 验签 vs access token + userinfo：

  原始审计方案：前端发送 Google ID token → Worker 本地验签 → 取 sub
  当前实际情况：GIS (Google Identity Services) implicit flow 返回的是
  access token，不是 ID token。要获取 ID token 需要：
    a. 使用 authorization code flow（需后端）
    b. 或在 GIS 配置中请求 id_token scope + 设置 nonce

  因此我建议分两步走：

  第一步（快速落地）：
  - 前端在请求头中发送 access token（Authorization: Bearer xxx）
  - Worker 调用 Google tokeninfo/userinfo API 验证 token 有效性 + 取 sub
  - 代价：每次 AI 请求多一次 Google API 调用（~50ms）
  - 收益：立即可实现，无需改前端 OAuth 流程

  第二步（后续优化，可选）：
  - 前端 GIS 配置添加 id_token scope
  - 获取 ID token 后，Worker 本地验签（用 Google JWKS 公钥）
  - 代价：需要改前端 OAuth 配置 + Worker 实现 JWT 验签
  - 收益：消除对 Google API 的运行时依赖，延迟更低

  为什么没有直接采用 ID token 方案：
  当前 GIS 配置只请求 access token，切换到 ID token 需要同时修改
  前端和 Worker，涉及 scope 变更、token 解析逻辑变更、JWKS 缓存等。
  在"快速关闭安全漏洞"的优先级下，access token + userinfo 方案
  可以在不动前端的情况下，仅通过 Worker 端改动就完成身份验证。
