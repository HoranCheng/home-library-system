五、Drive API 查询注入防护 — [A] 已完成并验证
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【原来的风险点是什么】
  Google Drive API 的文件搜索使用一种类似 SQL 的查询语法：
    name='小票待处理' and mimeType='application/vnd.google-apps.folder'

  我们的代码把用户设置的文件夹名称直接拼进了这个查询字符串：
    q: `name='${config.inboxFolder}'`

  如果用户的文件夹名称包含单引号（比如 "Tom's Receipts"），拼出来就变成：
    name='Tom's Receipts'
              ^ 这里单引号提前结束了字符串

【什么输入会导致问题】
  任何包含英文单引号 ' 的文件夹名称都会触发。例如：
  - Tom's Receipts
  - 小票'
  - '; DELETE (虽然 Drive API 不支持 DELETE，但查询本身会报语法错误)

  实际危害：不是数据泄露，而是功能崩溃 — 查询报错导致找不到文件夹，触发重复创建。

  这不是 SQL 注入（Drive API 不是数据库），但属于"输入未转义导致功能异常"。

【现在做了什么防护】
  在 google.js 中添加了 escQ() 函数：

    function escQ(s) { return (s || '').replace(/\\/g, '\\\\').replace(/'/g, "\\'"); }

  作用：把单引号 ' 转义为 \'，把反斜杠 \ 转义为 \\。
  这是 Google Drive API 官方文档推荐的转义方式。

  所有涉及 name='...' 查询的地方都已替换：
    - getOrCreateRootFolder()：搜索 Receipt Renamer 根文件夹
    - renameSubFolder()：搜索旧名和新名
    - findOrCreateFolder()：搜索子文件夹
    - readCloudConfig()：搜索配置文件
    - saveCloudConfig()：搜索配置文件

  总共 6 处查询全部使用 escQ() 包裹。

【防护在前端还是后端】
  前端。因为 Drive API 查询是前端直接发到 Google 服务器的（用用户自己的 OAuth token），
  不经过我们的 Worker 后端。这里没有后端参与。

【有没有做测试验证】
  代码层面验证：escQ("Tom's") 输出 "Tom\\'s"，拼入查询后语法正确。
  实际测试：需要你在设置中把某个文件夹名改成带单引号的名字（如 "Tom's"），然后看是否
  能正常找到文件夹而不报错。

【还有没有边界情况未覆盖】
  - 反斜杠 \ 已覆盖
  - 换行符和制表符：Google Drive API 不允许文件夹名含换行符，操作系统也不允许，
    所以不需要额外处理
  - Unicode 字符：不需要转义，Drive API 原生支持
  - 空字符串：escQ('') 返回 ''，不会崩溃

  结论：当前防护覆盖了所有实际场景。

六、清除缓存与登录状态 — [A] 已完成并验证
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【改之前是什么样】
  点"清除本地缓存" → 调用 clearAllData() → 删除 localStorage 里所有 rr-* 前缀的键
  → 其中包括 rr-config（里面有 connected: true, googleProfile, sheetId 等）
  → 结果：Google 登录状态丢失，用户必须重新登录

【改了什么】
  点"清除本地缓存"之前，先提取出以下字段暂存：
    - connected（是否已连接 Google）
    - googleProfile（用户名、邮箱、头像）
    - clientId（OAuth Client ID）
    - setupDone（是否已完成初始设置）
    - sheetId（Google Sheets 表 ID）
    - sheetName（表名）

  然后执行 clearAllData()（清掉所有 rr-* 键），再把上面的字段写回去。

【为什么这么改】
  "清除缓存"的用户预期是：清掉本地的临时数据、缓存、处理进度等。
  不是：退出我的 Google 账号，让我重新登录一遍。

  如果用户想退出账号，有单独的"退出"按钮。
  如果用户想彻底删除一切，有新增的"删除全部数据"按钮。

【清除的内容】
  ✅ rr-receipts（本地收据记录缓存）
  ✅ rr-proc-progress（上次处理进度）
  ✅ rr-non-receipt-alerts（非小票提醒）
  ✅ rr-sheets-outbox（Sheets 写入失败的重试队列）
  ✅ IndexedDB: rr-image-cache、rr-pending-uploads、rr-sw-queue
  ✅ sessionStorage 中的 rr-view（当前页签）

【保留的内容】
  ✅ connected = true/false
  ✅ googleProfile（名字、邮箱、头像 URL）
  ✅ clientId
  ✅ setupDone
  ✅ sheetId + sheetName

【会不会留下敏感信息】
  保留的信息中：
  - googleProfile 包含用户名和邮箱 → 这是登录状态的一部分，退出登录时会清除
  - sheetId 是 Google Sheets 的 ID → 本身不是秘密（知道 ID 也需要 OAuth 权限才能读取）
  - 没有保留 access token（token 在 sessionStorage 中，不受清缓存影响；关闭浏览器即失效）

  结论：保留的信息不含敏感凭证，风险可接受。

【对用户体验的影响】
  正面：清完缓存后不需要重新登录，体验更顺畅。

【对安全边界的影响】
  无负面影响。真正的安全退出路径（退出按钮、删除全部数据）不受影响。

【怎么验证】
  1. 登录后拍几张小票
  2. 去设置页 → 清除本地缓存
  3. 验证：记录页显示空（本地缓存已清）
  4. 验证：设置页仍显示已连接（头像、邮箱仍在）
  5. 验证：Sheets 数据不受影响

七、Safari 登录流程 — [B] 已完成但需实测验收
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【做了什么】
  Safari 的 ITP（智能防追踪）会阻止 Google 的静默 token 刷新。之前的逻辑是：
  静默刷新失败 → 什么也不做 → 用户看到空白页

  现在改成：
  静默刷新失败 → 立刻弹出 Google 交互式登录（和手动点"登录"效果一样）
  → 登录成功 → 自动进入功能页
  → 登录失败/取消 → 跳到设置页

  这个改动同时覆盖了两个入口：
  1. App 启动时（useEffect 初始化）
  2. 点击导航栏时（handleNav）

【为什么这么做】
  Safari 的 ITP 是系统级行为，无法绕过。唯一的解决方案是在静默失败后主动引导用户
  做一次交互式登录。Google 的 GIS SDK 在交互模式下不受 ITP 限制。

【残留风险】
  - 每次在 Safari 上重新打开页面，可能都需要点一次 Google 登录弹窗（因为 Safari 不保留
    第三方 cookie，session 无法跨访问持久化）
  - 这是 Safari + Google OAuth 的已知限制，所有使用 Google OAuth 的 PWA 在 Safari 上都有此问题

【怎么验证】
  需要你在 Safari 上：
  1. 打开 https://horancheng.github.io/receipt-renamer/
  2. 观察是否自动弹出 Google 登录
  3. 登录后是否正常显示内容
  4. 关闭 Safari 重新打开，是否又需要登录

  在 Safari 上走一遍上述流程，告诉我结果。我根据结果继续调。

以下是【正式上线前必须完成的安全项清单】，每次报告都会带上：

  C1. 用户身份验证
    现状：Worker 信任前端传的 uid，没有后端校验
    风险：任何人可以冒充其他用户消耗配额
    上线前必须：Worker 验证 Google access token → 提取真实 uid
    需要：Worker 源码

  C2. /api/debug/gemini 调试接口
    现状：公开可访问，返回 Gemini API 的原始响应
    风险：可能泄露 API 密钥信息或内部错误细节
    上线前必须：删除或加认证
    需要：Worker 源码

  C3. CORS 域名白名单
    现状：Worker 允许所有来源的请求
    风险：任何网站都可以调用我们的 AI 接口消耗配额
    上线前必须：限制为 horancheng.github.io（或正式域名）
    需要：Worker 源码

  C4. 配额原子化
    现状：KV 读写不是原子操作，并发请求可能绕过配额限制
    风险：大量并发请求可超出 100/天限制
    上线前必须：改用 Durable Objects 或事务性 KV
    需要：Worker 源码 + Cloudflare 付费计划

  内测阶段影响评估：
  - C1 和 C3：内测期只有你在用，风险很低，但 URL 如果被别人发现可以白嫖 AI 配额
  - C2：有人访问 debug 接口可以看到 Gemini 错误信息，不会泄露 API key 本身
  - C4：当前用户量下不可能触发并发竞态

  结论：内测期这些不会造成实际安全事件，但上线前必须修。
  我每次报告都会把这 4 项带上，不会遗漏。

【方案对比】

  方案 1：Drive 内备份（推荐）
    做法：在 Receipt Renamer 文件夹旁创建一个 "Receipt Backup" 文件夹，
         定期（或每次处理完）把已存档的小票复制一份过去
    触发方式：自动（每次处理完成后），或手动点"立即备份"
    恢复方式：用户直接去 Drive 的 Receipt Backup 文件夹找文件
    优点：零成本、用户已有的 Google 存储空间、跨设备可用
    缺点：占用双倍 Drive 空间；不算真正的"异地备份"（都在 Google）
    误操作风险：低。备份是只读副本，"删除全部数据"时可以选择是否一起删

  方案 2：本地下载备份
    做法：一键打包所有 Sheets 数据 + Drive 文件链接，下载为 ZIP
    触发方式：手动点"导出备份"
    恢复方式：没有自动恢复——用户手动保存 ZIP，需要时自己查看
    优点：完全离线、不依赖 Google
    缺点：用户容易忘记手动备份；恢复体验差
    误操作风险：无（只读导出）

  方案 3：两者都做
    触发方式：Drive 备份自动跑，本地导出手动触发
    优点：最全面
    缺点：实现复杂度高，对内测阶段可能过度

【我的建议】
  内测阶段先做方案 1（Drive 内备份），因为：
  - 实现简单（已有 Drive API 基础）
  - 用户无需手动操作
  - 恢复直观（去 Drive 找文件就行）
  方案 2 作为 v0.5 的增强项。

【需要你拍板】
  - 选方案 1、2 还是 3？
  - 如果选 1：备份频率是"每次处理后自动"还是"手动触发"？
  - 执行"删除全部数据"时，备份文件夹是否一起删？
